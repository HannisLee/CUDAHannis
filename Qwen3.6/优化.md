基于 vLLM benchmark 对 Qwen3.6-35B-A3B 在单卡环境下进行 serving 性能画像分析。构造 normal、prefill-heavy、decode-heavy 三类 workload，发现 512→128 场景在 4 req/s 下稳定运行，TTFT 约 114ms、TPOT 约 20.6ms；512→1024 场景在 2 req/s 下出现明显排队，TTFT 上升至 13.2s，说明 decode token 生成吞吐成为主要瓶颈；8192→128 场景接近 prefill 吞吐上限，并出现 decode inter-token latency 抖动。

  2. Stage 3 应该选哪些点？

你不需要把 Stage 2 的 14 个点全部 profile。那样 trace 文件会很大，而且不好分析。

应该只选代表性点。

必选点 1：normal 稳定点
input_len = 512
output_len = 128
request_rate = 4

意义：

这是健康状态下的 baseline。

你要看：

正常情况下 GPU timeline 是什么样？
kernel 是否连续？
TTFT 低的时候 CPU/GPU 协作是什么状态？
decode loop 是否稳定？

这个点是你后面所有 trace 的参照物。

必选点 2：normal 过载点
input_len = 512
output_len = 128
request_rate = 8

意义：

Stage 2 里 qps=8 的 TTFT 明显爆炸。你要看为什么。

重点问题：

GPU 是不是已经满了？
请求是不是在队列里排队？
prefill 是否成批堆积？
decode 阶段是不是被 batch 扩大后拖慢？
GPU timeline 有没有空洞？

它和 normal qps=4 对比，能告诉你：

同样 512→128，为什么 qps=4 稳定，qps=8 不稳定？
必选点 3：prefill 临界点
input_len = 8192
output_len = 128
request_rate = 1.5

意义：

这个点是长输入压力刚开始明显影响 decode 的位置。

重点看：

长 prefill 是否占用大块 GPU 时间？
decode token 之间是否被 prefill 插入打断？
prefill 是否导致 ITL 抖动？
max_num_batched_tokens 是否可能过大或过小？
是否值得开启 / 调整 chunked prefill？

这个点的学习价值非常高，因为它能让你理解：

长上下文不是只影响 TTFT，也可能影响 decode 的稳定性。

必选点 4：decode 临界点
input_len = 512
output_len = 1024
request_rate = 0.75

意义：

这个点接近长输出场景的 decode 上限，但还没明显崩。

重点看：

decode loop 是不是连续？
每 token 生成的 kernel pattern 是什么？
paged attention / GEMM / sampling 谁占主要时间？
CUDA Graph replay 是否稳定？
GPU 是否已经接近满载？

这个点能帮你理解：

长输出场景的吞吐上限到底由什么决定。

可选点 5：decode 过载点
input_len = 512
output_len = 1024
request_rate = 1.0

意义：

这个点是 qps=0.75 和 qps=1.25 之间的关键过渡点。

你要看：

TTFT 为什么从百毫秒级上升到秒级？
是 GPU 算不过来，还是请求排队？
TPOT 有没有明显恶化？
decode kernel 是否变得更密集？
max_concurrent_requests 是否导致 scheduler 压力？

如果时间够，这个点非常建议抓。
